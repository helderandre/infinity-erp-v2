Diagnostico

5 lacunas concretas

1. Documentos nao fluem

Lead faz upload de CC → lead_attachments (isolado). Quando vira owner, consultor pede os mesmos documentos novamente.

2. Leads duplicados

Sem verificacao de NIF/email. Joao pode existir 3 vezes na tabela leads com dados diferentes.

3. Temperatura e global

Um lead tem um estado, uma temperatura. Impossivel ser quente para Lisboa e frio para Porto na mesma campanha.

4. Sem rastreabilidade Lead → Owner

Sem FK entre tabelas. Saber se um lead e o mesmo que um owner requer lookup manual por NIF.

5. Negocios: 63 colunas a misturar 3 conceitos

Criterios de busca (orcamento, quartos_min) + Dados de imovel (preco_venda, area) + Perfil financeiro (credito, capital) — tudo na mesma tabela.

Lacuna #1 Resolvida

Fluxo de documentos — PROPOSTO

Lead Joao faz upload CC + Comp. Morada + IRS
↓
doc_registry — contact_id = Joao
TODOS os documentos num so lugar, desde o primeiro upload
↓

Mesmo contact_id, sem copiar dados
↓

Encontra TODOS os docs — uploaded como lead ou owner

Zero re-uploads. Zero trabalho duplicado.

Decisao

Onde esta o over-engineering real?

ESTRUTURA ACTUAL

✗ negocios: 63 colunas a misturar criterios de busca + dados de imovel + perfil financeiro

✗ leads + owners: 8 grupos de campos identicos duplicados

✗ lead_attachments: sistema paralelo de documentos que nao conecta a nada

✗ temperatura: campo fixo no lead — forca estado global

vs

PROPOSTA

✓ Separa conceitos: identidade vs papel vs engagement vs intencao

✓ Single source: uma pessoa = um registo, zero duplicacao

✓ Documentos fluem: upload uma vez, reutilizar em todos os processos

✓ Contextual: estado, temperatura e fonte por campanha
